# 조회수 집계 확장 메모

## 현재 방식

- 현재 `src/lib/data/content.ts`는 `events` 테이블에서 `content_view` 로그를 읽어온 뒤 서버 메모리에서 콘텐츠별 조회수를 합산합니다.
- MVP와 초기 운영 단계에서는 구조가 단순하고 구현 비용이 낮아 현실적인 선택입니다.

## 장기 리스크

- 이벤트 로그가 커질수록 조회수 계산 시 읽어야 하는 데이터가 빠르게 늘어납니다.
- 트래픽이 많아지면 응답 시간과 서버 메모리 사용량이 함께 증가할 수 있습니다.
- 홈, 목록, 검색 등 여러 페이지가 같은 집계 경로를 공유하고 있어 병목이 생기기 쉽습니다.

## 후속 개선 방향

- Supabase RPC로 DB 내부 집계 쿼리 수행
- materialized view로 조회수 요약 결과 캐시
- 콘텐츠별 counter cache 또는 별도 summary table 운영
- edge function / cron / background job 기반 비동기 집계

## 이번 차수 결정

- 3.9차에서는 사용자 첫인상과 UI 품질 보정이 우선이므로 집계 구조는 대공사하지 않습니다.
- 대신 현재 방식이 기술 부채라는 점을 코드 주석과 이 문서에 명시하고, 후속 차수의 집계 고도화 과제로 남깁니다.
